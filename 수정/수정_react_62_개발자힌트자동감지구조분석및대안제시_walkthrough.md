# 개발자 힌트 자동 감지 구조 분석 및 대안 제시 결과서 (Walkthrough)

현재 적용되어 있는 중앙 집중식 개발자 힌트(DevHints) 매핑 방식의 잠재적 누락 문제를 예방하기 위해, 런타임 자동 감지 구현의 기술적 한계를 분석하고 이를 극복할 수 있는 실무적 대안을 정리하였습니다.

## 1. 런타임 자동 감지(Dynamic Detection)의 한계 분석
- **의존성 분석의 한계**:
  * 브라우저 런타임(React/Next.js)에서 현재 화면이 의존하는 파일 리스트(fe/be)와 관련 SQL 테이블을 동적으로 감지하는 것은 보안 샌드박스 및 소스코드 빌드 컴파일 구조상 어렵습니다.
  * API 호출은 네트워크 수준(Fetch/Axios)에서 트래킹할 수 있으나, 해당 API가 백엔드 내부에서 어떤 Python 스크립트와 SQLite DB 테이블을 경유하여 쿼리하는지까지 프론트엔드가 실시간 감지하여 받아오는 것은 추가적인 미들웨어 로깅 로직이 필요하여 과도한 리소스를 요구(오버엔지니어링)합니다.

## 2. 누락 방지를 위한 2대 대안 구조

### 대안 A: 페이지 개별 직접 선언 및 렌더링 방식 (추천)
- **개념**: 중앙의 `DevHints.js` 파일 내 큰 맵을 없애고, 각 페이지 파일(`page.js`) 내부 하단에 직접 해당 페이지의 힌트 데이터를 props로 렌더링하는 구조입니다.
  * 예: `fe/src/app/dashboard/jobs/page.js` 파일 하단에 `<DevHints fe={['jobs/page.js', 'useJobs.js']} be={['/api/jobs']} db={['tb_jobs']} />` 배치.
- **장점**: 페이지가 추가/수정될 때 해당 소스 코드 파일 자체에서 힌트가 바로 선언 및 변경되므로, 별도의 맵 파일을 찾아 갱신해야 하는 수고를 덜고 누락을 완벽히 방지할 수 있습니다.

### 대안 B: 백엔드 Swagger/OpenAPI 메타데이터 동적 동기화 방식
- **개념**: 백엔드의 각 API 라우트 장식자(Decorator)에 관련 DB 테이블 및 매핑 스크립트 메타데이터를 기입해 두고, 프론트엔드 힌트 컴포넌트가 마운트될 때 현재 주소와 매핑되는 백엔드 API를 쿼리하여 정보를 가져오는 구조입니다.
- **장점**: 프론트엔드와 백엔드의 메타데이터가 백엔드 라우터 소스코드 하나에 집중되므로 일치성이 높아집니다.
